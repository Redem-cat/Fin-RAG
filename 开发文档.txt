================================================================================
FinRAG-Advisor 基于多模态知识图谱增强的智能投顾与合规双模 RAG系统
================================================================================

## 项目概述

一个面向金融机构的，不仅支持客户与员工的自然语言问答，而且深入融合了
投资建议生成与合规风险自动校验，实现智能服务＋自动合规审查一体化。

## 核心创新点

1. **双RAG子系统**：同时运行投资建议生成与合规验证两个RAG子系统，
   合规验证对投资建议生成进行实时审计，降低幻觉

2. **多模态知识图谱增强**：将年报等的表格图片通过OCR或多模态大模型
   结构化解析转化为知识三元组，构建动态金融KG

3. **实时监管更新**：接入央行证监会等RSS源，自动抓取新政策与更新

4. **用户意图分层**：将用户意图进行合理的细分，提升准确率与效率

5. **文档处理**：使用PyPDF2 + LayoutRecognizer进行布局识别并区分各种
   文档结构；表格使用TableStructureRecognizer还原为HTML或JSON

6. **智能分块策略**：层级感知动态合并算法

7. **混合检索**：
   - 向量检索：使用BGE-M3-Financial模型
   - 关键词检索：选用Elasticsearch
   - 知识图谱检索：使用基于Neo4j查询（某基金→所属公司→近期处罚记录）
   - 结果融合：通过RRF融合三种结果

8. **系统化评估**：使用RAGAS框架检验五大指标：
   - Faithfulness（忠实度）
   - Context Precision（上下文精确度）
   - Answer Relevance（答案相关性）
   - Response Time（响应时间）
   - Compliance Coverage（合规覆盖度）
   并进行可视化仪表盘展示

# Streamlit 进程管理问题（2026-03-02）
--------------------------------------------------------------------------------

## 问题描述
刷新页面或切换 CHAT/EVAL 时，Ollama 和 Elasticsearch 连接没有正确关闭，
导致：
1. 旧进程堆积，占用端口和内存
2. 应用卡住无法响应
3. 需要手动杀掉进程才能恢复

## 根本原因
1. Ollama 的 HTTP 连接池没有正确关闭
2. Elasticsearch 连接没有正确释放
3. LangChain/Ollama 库的连接管理机制不完善
4. Streamlit 的页面刷新机制（重新执行全部代码）与全局状态冲突

## 当前尝试方案（效果不佳）
1. 在 `src/rag.py` 中添加 `cleanup_resources()` 函数
   - 关闭 Elasticsearch 连接
   - 清理 Ollama 客户端
   - 重置全局变量

2. 在 `src/streamlit_app.py` 中添加页面切换清理逻辑
   - 跟踪当前页面状态
   - 检测页面切换时调用 `cleanup_resources()`

**结果：** 效果不明显，资源泄漏仍然存在

## 待解决的方案（按优先级）
### 方案1：强制重启（最彻底）
- 页面刷新时自动杀掉所有相关进程（Python、Ollama、Streamlit）
- 使用子进程启动并监控，检测到刷新时 kill 并重启
- **优点**：彻底清理，100% 有效
- **缺点**：用户体验差，会有短暂黑屏

### 方案2：独立服务架构
- 将 RAG 核心逻辑移到独立的 FastAPI 服务
- Streamlit 只做前端 UI，通过 HTTP API 调用
- 后台服务独立运行，不受 Streamlit 刷新影响
- **优点**：架构清晰，稳定性高
- **缺点**：需要重构，增加部署复杂度

### 方案3：使用 Streamlit Experimental Features
- 使用 `@st.experimental_singleton` 缓存全局连接
- 使用 `st.runtime.scriptrunner` 的生命周期钩子
- **优点**：无需重构，官方方案
- **缺点**：实验性功能，可能不稳定

### 方案4：连接池管理优化
- 为 Ollama 和 Elasticsearch 添加显式连接池
- 使用 `with` 上下文管理器确保资源释放
- 添加超时和健康检查
- **优点**：侵入性小
- **缺点**：治标不治本

# 资源清理方案（2026-03-02）
--------------------------------------------------------------------------------

## 问题描述
系统运行时按 `Ctrl+C` 退出时，Ollama 和 Elasticsearch 连接没有正确关闭，
导致：
1. 旧进程堆积，占用端口和内存
2. 应用卡住无法响应

## 已实现方案

### 方案二：批处理启动脚本（已实现）✅

**文件位置**: `scripts/run_with_cleanup.bat`

**功能特点**:
- 自动启动 Streamlit 应用
- 检查依赖服务状态
- 按 Ctrl+C 后自动清理资源
- 清理 Streamlit 和 Python 进程
- 可选清理 Ollama（默认保留）

**使用方法**:
```bash

scripts\run_with_cleanup.bat
```

**脚本流程**:
1. 检查 .env 文件和依赖服务
2. 启动 Streamlit 应用（端口 8501）
3. 用户按任意键停止
4. 自动清理所有相关进程
5. 显示清理结果

**优点**:
- 简单易用，一行命令启动
- 自动清理，无需手动杀进程
- 可选择是否清理 Ollama
- 清晰的提示信息

**注意事项**:
- Ollama 默认不清理，可取消注释开启
- 清理会停止所有 Python 进程（除了本脚本）
- 需要手动按任意键触发清理流程

### 方案一：Python 包装脚本

**核心代码**:
```python
# scripts/run_with_cleanup.py
import sys
import subprocess
import atexit
import signal
from pathlib import Path

PROJECT_ROOT = Path(__file__).parent.parent

def cleanup_processes():
    """清理所有相关进程"""
    print("\n正在清理资源...")
    try:
        subprocess.run(["taskkill", "/F", "/IM", "ollama.exe", "/T"],
                      capture_output=True)
        subprocess.run(["taskkill", "/F", "/IM", "python.exe", "/T"],
                      capture_output=True)
        print("✓ 清理完成")
    except Exception as e:
        print(f"⚠ 清理时出错: {e}")

def signal_handler(signum, frame):
    """信号处理"""
    cleanup_processes()
    sys.exit(0)

def run_streamlit():
    """启动 Streamlit"""
    try:
        cmd = [
            sys.executable, "-m", "streamlit", "run",
            str(PROJECT_ROOT / "src" / "streamlit_app.py"),
            "--server.port", "8501",
            "--server.headless", "true"
        ]
        proc = subprocess.Popen(cmd)
        proc.wait()
    except KeyboardInterrupt:
        print("\n检测到中断信号...")
    finally:
        cleanup_processes()

if __name__ == "__main__":
    signal.signal(signal.SIGINT, signal_handler)
    signal.signal(signal.SIGTERM, signal_handler)
    atexit.register(cleanup_processes)
    run_streamlit()
```

**优点**:
- 跨平台兼容（Windows/Linux）
- 自动捕获 Ctrl+C 信号
- 更精确的进程管理

## 1. 修复合规审查器导入错误

**问题描述**:

```
警告: 合规审查器初始化失败: name 'ComplianceChecker' is not defined
```

**根本原因**:
`from src.compliance_checker import ComplianceChecker` 导入语句放在 try 块内部，
如果导入失败，try 块中的后续代码不会执行，导致 `ComplianceChecker` 未定义。

**修复方案**:
将导入语句移到 try 块的第一层，并添加 `ImportError` 捕获：

```python
# 修复前（错误）
try:
    from src.compliance_checker import ComplianceChecker
    ComplianceChecker()  # 导入失败时无法执行

# 修复后（正确）
try:
    from src.compliance_checker import ComplianceChecker
    try:
        if not hasattr(_get_compliance, 'instance'):
            _get_compliance.instance = ComplianceChecker()
        # ... 使用 ComplianceChecker

    except ImportError as e:
        # 导入失败的处理
        compliance_result = {...}

except ValueError as e:
    # API Key 未配置
    compliance_result = {...}

except Exception as e:
    # 其他异常
    compliance_result = {...}
```

**修改文件**:
- `src/rag.py:865-929` - `ask_question()` 函数中的合规审查
- `src/rag.py:1081-1119` - `stream_ask_question()` 函数中的合规审查

**修复效果**:

## 5. Context Recall 指标说明

**指标含义**:
衡量检索到的上下文是否包含了标准答案（ground_truth）中的所有关键信息。

**计算公式**:
```
Context Recall = (检索到的上下文中包含的关键信息数量) / (标准答案中的总关键信息数量)
```

**为什么一直等于 1**:
1. 检索系统质量好，召回了所有必要信息
2. 测试集问题设计简单，标准答案在知识库中完整存在
3. 这是好事，说明检索系统覆盖能力强

**判断标准**:
- Recall = 1 + Precision 高 = 完美检索
- Recall = 1 + Precision 中 = 覆盖全面但有噪音（可优化精排）
- Recall 低 = 遗漏关键信息（需要优化召回）

## 功能概述

合规审查模块用于对AI生成的投资建议进行实时合规性检查，确保回答符合
金融监管要求，避免违规内容。

## 核心组件

### 1. ComplianceChecker (`src/compliance_checker.py`)

**功能**:
- 使用 DeepSeek API 对生成的回答进行合规性审查
- 返回合规判断结果及详细的风险分析

**初始化**:
```python
from src.compliance_checker import ComplianceChecker

checker = ComplianceChecker()  # 需要配置 DEEPSEEK_API_KEY
```

**主要方法**:

| 方法 | 说明 | 返回值 |
|------|------|--------|
| `check(question, answer)` | 审查问答对 | `Dict[str, Any]` |

**返回字段**:
```python
{
    "is_compliant": bool,      # 是否合规
    "risk_level": str,         # 风险等级（low/medium/high/unknown）
    "violations": list,        # 违规点列表
    "suggestions": str,        # 改进建议
    "summary": str,            # 合规总结
    "raw_response": str        # API原始返回
}
```

### 2. 集成到RAG流程

**调用位置**:
- `src/rag.py` `ask_question()` 函数
- `src/rag.py` `stream_ask_question()` 函数

**调用时机**:
LLM生成答案后立即进行合规审查

**代码示例**:
```python
# 生成LLM答案
answer = llm.invoke(...)

# 合规审查
if COMPLIANCE_ENABLED and compliance_checker:

return {
```

## 合规检查规则

**检查内容**:
1. 是否包含具体的投资建议（买入/卖出建议）
2. 是否保证收益率或承诺收益
3. 是否夸大宣传或误导性信息
4. 是否符合金融广告法规
5. 是否适当性管理（风险提示）
6. 是否存在敏感违规内容

**风险等级判断**:
- `low` - 无明显违规风险
- `medium` - 存在轻微风险，建议修改
- `high` - 存在明显违规，必须修改
- `unknown` - 无法判断（API错误或特殊情况）

## 错误处理

**可能的错误**:
1. **ImportError**: 模块导入失败
   - 检查 `compliance_checker.py` 是否存在

2. **ValueError**: API Key 未配置或无效
   - 检查 `.env` 文件中的 `DEEPSEEK_API_KEY`

3. **ConnectionError**: API连接失败
   - 检查网络连接和API服务状态

4. **JSONDecodeError**: 返回格式错误
   - 查看 `compliance_checker.py` 日志输出

**降级策略**:
- 合规审查失败时，不影响主流程
- 返回 `compliance = None`
- 前端显示 `[未知]` 标识
