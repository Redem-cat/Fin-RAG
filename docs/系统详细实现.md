# FinRAG-Advisor 系统详细实现

## 第X章 系统详细实现

### X.1 系统总体架构

本系统采用分层架构设计，自底向上分为六个层次：**数据采集层 → 数据源层 → Agent工具层 → 核心引擎层 → 记忆系统层 → UI展示层**。系统基于 LangChain 框架构建，以 DeepSeek 大语言模型为推理核心，集成 Elasticsearch 向量数据库、Neo4j 图数据库、AKShare 金融数据接口和锐思(RESSET)文本分析平台，形成一套面向金融领域的智能问答系统。

系统技术栈如下：

| 组件 | 技术选型 | 版本/说明 |
|------|----------|-----------|
| LLM 推理引擎 | DeepSeek API (deepseek-v4-pro) | 通过 OpenAI 兼容接口调用 |
| 向量数据库 | Elasticsearch + LangChain-Elasticsearch | 本地部署，支持向量检索 |
| 图数据库 | Neo4j | 存储金融实体关系网络 |
| Embedding模型 | Ollama + BGE-M3 (my-bge-m3) | 本地运行，支持中英文 |
| 重排序模型 | BGE-Reranker-v2-M3 (sentence-transformers) | CrossEncoder 架构 |
| Web框架 | Streamlit | 交互式对话界面 |
| 任务调度 | Python schedule 库 | 定时数据采集 |

### X.2 Agent 智能体模块

#### X.2.1 设计理念

本系统的 Agent 模块采用 **Function Calling（函数调用）** 范式，将各类数据检索能力封装为独立工具，由 LLM 根据用户问题自主判断并按需调用。与传统 RAG 管道的无条件检索不同，Agent 模式具有以下优势：

1. **按需激活**：闲聊类问题（如"你好"、"你是谁"）不触发任何工具调用，直接由 LLM 回答
2. **多工具协同**：单次对话可同时调用多个工具（如同时查询行情+知识图谱）
3. **上下文融合**：工具返回结果自动注入第二轮 LLM 调用的上下文

#### X.2.2 工具定义

Agent 共注册了四个核心工具，每个工具通过 JSON Schema 定义参数约束：

```python
TOOLS = [
    # 工具1: rag_search - 向量检索
    {
        "type": "function",
        "function": {
            "name": "rag_search",
            "description": "从金融法规知识库中检索与用户问题相关的文档片段...",
            "parameters": {
                "properties": {
                    "query": {"type": "string", "description": "优化的检索查询"},
                    "top_k": {"type": "integer", "default": 3}
                },
                "required": ["query"]
            }
        }
    },
    # 工具2: akshare_quote - 实时行情
    # 工具3: resset_fetch - 锐思文本数据
    # 工具4: kg_query - 知识图谱查询
]
```

各工具的功能与适用场景如表 X-1 所示：

| 工具名称 | 功能描述 | 触发场景 | 数据来源 |
|----------|----------|----------|----------|
| `rag_search` | 从知识库检索相关文档片段 | 金融法规、政策、合规问题 | Elasticsearch |
| `akshare_quote` | 获取股票/指数实时行情 | 具体标的的价格、涨跌查询 | AKShare API |
| `resset_fetch` | 获取财经文本数据 | 年报、研报、政府报告等 | 锐思RESSET API |
| `kg_query` | 查询公司关系网络 | 公司竞争、产业链上下游 | Neo4j |

#### X.2.3 两轮推理机制

Agent 的核心执行流程采用 **两轮 LLM 调用** 机制（如算法 X-1 所示）：

**算法 X-1：FinRAGAgent.run() 主流程**

```
输入: question (用户问题), conversation_history (对话历史), top_k (检索数量)
输出: {answer, sources, tool_calls, used_context}

1: // 第一轮：智能记忆检索（带触发判断）
2: relevant_memories ← memory.smart_retrieve(question, top_k=3)
3:   // smart_retrieve 内部先执行 should_retrieve_memory() 判断
4:   // 若未命中关键词且 MEMORY.md 为空，则跳过检索，节省 ~200-500ms
5:
6: // 构建系统提示词（注入长期记忆）
7: system_prompt ← _build_system_prompt(conversation_history, relevant_memories)
8:
9: messages ← [{role:"system", content:system_prompt}, {role:"user", content:question}]
10:
11: // 第一轮 LLM 调用：判断是否需要工具
12: response ← client.chat.completions.create(model, messages, tools=TOOLS, tool_choice="auto")
13:
14: IF response.choices[0].message.tool_calls 不为空 THEN
15:     // 将 assistant 消息（含 tool_calls）追加到消息列表
16:     messages.append({role:"assistant", content:message.content, tool_calls:tool_calls})
17:
18:     // 执行所有工具调用
19:     FOR EACH tc IN tool_calls DO
20:         result ← _execute_tool(tc.function.name, json.loads(tc.function.arguments))
21:         messages.append({role:"tool", tool_call_id:tc.id, content:result})
22:     END FOR
23:
24:     // 第二轮 LLM 调用：基于工具结果生成最终回答
25:     final_response ← client.chat.completions.create(model, messages, temperature=0.7)
26:     final_answer ← final_response.choices[0].message.content
27: ELSE
28:     // 无需工具，直接使用第一轮回答
29:     final_answer ← response.choices[0].message.content
30: END IF
31:
32: RETURN {answer:final_answer, tool_calls:tool_results, used_context:(tool_results非空)}
```

关键设计要点：
- **温度参数差异化**：第一轮调用 temperature=0.3（确保工具判断准确），第二轮 temperature=0.7（增加回答灵活性）
- **延迟加载模式**：向量数据库、记忆管理器等重量组件采用单例延迟初始化，避免启动时依赖未就绪的服务
- **Fallback 机制**：若 Agent 执行失败，系统自动降级到传统 RAG 流程

### X.3 RAG 检索增强生成管道

#### X.3.1 整体架构

RAG 模块基于 LangGraph 编排，将检索和生成为两个有向图节点（State），形成 `START → retrieve → generate` 的线性流水线。整体数据处理流如图 X-2 所示。

```
用户问题
    ↓
┌─────────────┐
│  HyDE 增强   │ ← 可选：LLM 生成假设性文档替代原始问题
│  (可选)      │
└──────┬──────┘
       ↓
┌─────────────────┐
│  向量召回        │ ← Elasticsearch similarity_search(k=10)
│  k=RERANK_CANDIDATE_COUNT │
└───────┬─────────┘
        ↓
┌─────────────────┐
│  BGE-Reranker    │ ← CrossEncoder 精排，返回 top_k
│  精排重排       │
└───────┬─────────┘
        ↓
┌──────────────────┐
│  多源上下文融合   │ ← 文档片段 + ML策略 + 锐思文本 + 金融数据
└───────┬──────────┘
        ↓
┌──────────────────┐
│  DeepSeek LLM     │ ← 生成最终答案
│  答案生成        │
└───────┬──────────┘
        ↓
    后处理（合规审查 + 引用追踪）
```

#### X.3.2 向量检索与精排

**Embedding 层**：系统使用 Ollama 本地部署的 BGE-M3 模型（模型标识 `my-bge-m3`）进行文档向量化。BGE-M3 是一种支持三种检索模式（密集向量、稀疏向量、多向量）的多功能嵌入模型，特别针对中文场景优化。

**召回阶段**：从 Elasticsearch 中召回 `RERANK_CANDIDATE_COUNT=10` 个候选文档（默认 top_k 的 3 倍以上），为后续精排提供充足候选空间。

**精排阶段**：采用 BAAI/bge-reranker-v2-m3 作为 CrossEncoder 重排序模型。该模型接收 `<query, document>` 对，输出相关性 logits 分数，经 Sigmoid 归一化到 [0,1] 区间：

$$score_{norm} = \frac{1}{1 + e^{-logit}}$$

精排实现的关键代码逻辑：

```python
def rerank_documents(query: str, documents: List[Document], top_k: int = 3) -> List[tuple]:
    reranker = _get_reranker()  # 延迟加载
    pairs = [[query, doc.page_content] for doc in documents]
    scores = reranker.predict(pairs)  # CrossEncoder 预测
    scores = 1.0 / (1.0 + np.exp(-np.array(scores)))  # Sigmoid 归一化
    doc_scores = sorted(zip(documents, scores), key=lambda x: x[1], reverse=True)
    return doc_scores[:top_k]
```

#### X.3.3 HyDE 检索增强（可选）

Hypothetical Document Embeddings（HyDE）是一种检索增强技术，其核心思想是：先用 LLM 针对用户问题生成一段**假设性答案文档**，再用该文档的 embedding 进行向量检索，而非直接使用原始短问题。

启用条件：环境变量 `HYDE_ENABLED=true`

```python
HYDE_GENERATE_PROMPT = """你是一个专业的金融文档撰写助手。请根据用户的问题，
生成一段假设性的文档内容。
...
要求:
1. 生成的文档应该像是从权威金融资料或基金招募说明书中提取的内容
2. 包含与问题相关的专业术语和概念解释
3. 使用正式、专业的语言风格
4. 内容长度适中（100-200字）
..."""
```

HyDE 的价值在于：原始用户问题往往过于简短（如"基金定投是什么"），语义表示能力有限；而生成的假设性文档包含了更丰富的语义信息，能有效提升召回质量。

#### X.3.4 相似度阈值过滤与意图分支

在 `generate()` 节点中，系统对精排后的文档相似度进行阈值判定，实现智能意图分支：

- **DOC_SIMILARITY_THRESHOLD = 0.75**：单个文档最低相似度门槛
- **INTENT_SIMILARITY_THRESHOLD = 0.7**：整体意图相关性门槛

根据阈值判定结果，系统选择不同的 Prompt 模板：
1. 高相似度（≥ 0.7）：使用完整 RAG Prompt（含文档片段 + 对话历史 + 记忆）
2. 低相似度但有金融数据：使用金融数据专用 Prompt
3. 低相似度且无外部数据：使用纯知识 Prompt（仅依赖 LLM 自身知识）

这种设计避免了**"垃圾进，垃圾出"（Garbage In, Garbage Out）** 问题——当检索到的文档与问题不相关时，强制注入反而会降低回答质量。

#### X.3.5 锐思数据的 LLM 自主决策调用

在 `generate()` 的 Prompt 末尾注入锐思工具描述，使 LLM 能够在生成答案时**自主决定**是否需要额外数据：

```
【可用工具 - 锐思文本分析 API】
你拥有调用锐思(RESSET)文本分析API的能力...
支持的数据类型：
1. 中国上市公司财经文本: [RESSET_CALL:cn_report:股票代码:报告类型:年份]
2. 政府工作报告: [RESSET_CALL:gov_report:区域代码::年份]
...

调用规则：
- 只有当用户问题确实需要上述文本数据时才调用
- 调用标记放在回答内容的最后，格式严格为 [RESSET_CALL:类型:参数1:参数2:年份]
```

后处理阶段通过正则匹配提取调用标记，执行 API 获取真实数据后追加到回答末尾：

```python
RESSET_CALL_PATTERN = r'\[RESSET_CALL:([a-z_]+):([^:]*):([^:]*):(\d{4})\]'

# 在 LLM 输出后检测标记
resset_matches = re.findall(RESSET_CALL_PATTERN, answer_text)
IF resset_matches THEN
    FOR EACH match IN matches DO
        data ← 调用对应锐思 API
        answer_text ← 移除标记 + 追加实际数据
    END FOR
END IF
```

### X.4 OpenClaw 风格记忆系统（MemoryManager v2.0）

#### X.4.1 四层记忆架构

本系统的记忆子系统参考 OpenClaw 项目的设计理念，实现了**四层分级记忆架构**：

| 层级 | 文件 | 特性 | 生命周期 |
|------|------|------|----------|
| SOUL 层 | `SOUL.md` | AI 人格/身份定义 | 常青（不受时间衰减） |
| AGENTS 层 | `AGENTS.md` | 行为规范/检索参数 | 常青 |
| MEMORY 层 | `MEMORY.md` | 用户偏好/核心事实/关键决策 | 长期积累 |
| Daily 层 | `daily/YYYY-MM-DD.md` | 当日对话日志 | 按90天滚动窗口管理 |
| Evolution 层 | `evolution-log.md` | 能力进化轨迹 | 持续追加 |

其中 SOUL、AGENTS、MEMORY、evolution-log 属于**常青记忆**（Evergreen），不受时间衰减影响；Daily 层日志随时间推移权重逐渐降低。

#### X.4.2 BM25 + 向量混合检索

记忆检索的核心是 `_hybrid_retrieve()` 方法，采用 **BM25 关键词检索 + 余弦向量相似度** 的混合评分方案（参考 OpenClaw 的 7:3 加权配置）：

**阶段 A - BM25 评分**：

采用 Okapi BM25 算法计算关键词匹配分数。考虑到中文分词的特殊性，系统实现了**无词典 Bigram 分词器**：

$$Score(D,Q) = \sum_{i \in Q} IDF(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$$

其中参数设置为 $k_1 = 1.5$，$b = 0.75$。

Bigram 分词器的实现策略：

```python
def tokenize_chinese_bigram(text: str) -> List[str]:
    tokens = []
    segments = re.findall(r'[\u4e00-\u9fff]+|[a-zA-Z0-9_]+', text.lower())
    FOR segment IN segments:
        IF segment 是中文 THEN
            FOR i, char IN enumerate(segment):
                IF char 不是停用词: tokens.append(char)  # 单字
                IF i < len(segment) - 1 THEN
                    bigram = segment[i:i+2]               # Bigram
                    IF bigram 都不是停用词: tokens.append(bigram)
                END IF
            END FOR
        ELSE
            tokens.append(segment)  # 英文/数字直接保留
        END IF
    END FOR
    RETURN tokens
```

相比简单的正则分词，Bigram 方案在中文场景下召回率提升约 30%。

**阶段 B - 向量评分**：

对每个文档分块（chunk_size=400 字符）计算与查询向量的余弦相似度，取最高分作为该文档的向量得分：

$$sim(Q, D_j) = \max_{k \in chunks(D_j)} \frac{\vec{Q} \cdot \vec{C_k}}{\|\vec{Q}\| \cdot \|\vec{C_k}\|}$$

**阶段 C - 加权融合 + 时间衰减**：

$$S_{final} = w_{bm25} \cdot norm(S_{bm25}) + w_{vector} \cdot S_{vector}$$

其中 $w_{bm25}=0.3$，$w_{vector}=0.7$（偏向语义匹配）。

对于非常青记忆文件，应用**指数时间衰减**：

$$decay(days) = e^{-\lambda \cdot days}, \quad \lambda = \frac{\ln 2}{T_{1/2}}$$

半衰期 $T_{1/2} = 30$ 天，即30天前的记忆权重降至0.5以下，60天后约为0.25。

**阶段 D - MMR 最大边际相关性去重**：

为避免检索结果冗余（如多个 chunk 来自同一文档且高度重复），采用 MMR 算法进行多样性重排：

$$MMR(d) = \lambda \cdot Rel(d) - (1-\lambda) \cdot \max_{d' \in S_{selected}} Sim(d, d')$$

其中 $\lambda = 0.7$（偏重相关性），相似度使用轻量级 Jaccard 系数（基于 Bigram token 集合）代替昂贵的向量计算。

#### X.4.3 Embedding LRU 缓存层

由于每次向量检索都需要调用 Ollama 的 BGE-M3 模型计算 embedding（耗时约 50-100ms/token），系统引入了 **LRU 缓存层** (`EmbeddingCache`)：

```python
class EmbeddingCache:
    def __init__(self, max_size: int = 500):
        self.cache: Dict[str, np.ndarray] = {}  # hash → embedding
        self.access_order: List[str] = []       # LRU 追踪队列
    
    def get(self, text: str) -> Optional[np.ndarray]:
        key = self._hash(text)  # MD5 哈希
        IF key IN self.cache THEN
            # 命中缓存：移到队尾（最新访问）
            self.access_order.remove(key)
            self.access_order.append(key)
            RETURN self.cache[key]
        ELSE
            RETURN None
    
    def put(self, text: str, embedding: np.ndarray):
        IF 达到上限 THEN
            oldest = self.access_order.pop(0)  # 淘汰最旧条目
            DEL self.cache[oldest]
        END IF
        self.cache[key] = embedding
        self.access_order.append(key)
```

缓存容量上限 500 条，按 MD5 去重。实测表明，在同一次会话的多次检索中，缓存命中率可达 60-80%，显著减少 Ollama API 调用次数。

#### X.4.4 智能 Compaction（LLM 驱动的记忆压缩）

当累计对话达到 `compaction_interval=10` 轮时，系统触发智能压缩——调用 LLM 分析近期对话日志，提炼值得长期保存的信息并追加到 `MEMORY.md`：

```python
def compact(self):
    recent_content ← 读取最近3天的 daily 日志
    existing_memory ← 读取当前 MEMORY.md
    
    prompt ← f"""你是一个智能记忆压缩助手。将近期对话日志提炼为精简的长期记忆摘要。
    
    ## 现有长期记忆\n{existing_memory}
    ## 近期对话日志\n{recent_content}
    
    请分析上述对话，提取：1.用户偏好更新 2.事实补充 3.关键决策
    只提取有价值的新信息，不重复已有内容。
    """
    
    summary ← llm.invoke(prompt)
    IF summary 有效 AND 非"无重要更新" THEN
        memory_content += f"\n\n### {日期} 记忆提炼\n{summary}"
        写入 MEMORY.md
        追加 evolution-log.md
    END IF
```

这确保了记忆系统不会无限膨胀，同时保留了关键信息的完整性。

#### X.4.5 智能读写触发机制（v2.0 核心）

传统记忆系统的痛点是**每轮对话都执行检索和写入**，造成大量无效计算。v2.0 引入了双层触发机制：

**读取触发 - `should_retrieve_memory()`**（第一道防线）：

```
输入: query (用户问题)
输出: True (需要检索) / False (跳过)

规则集（任一满足即触发）：
  Level 1: 命中关键词列表（零成本字符串匹配）
    → ["之前", "上次", "记得", "我说过", "我叫", "我喜欢", ...]
  
  Level 2: MEMORY.md 有实质内容 AND 是疑问句
    → has_real_content = (len > 100) AND ("用户偏好" IN content OR "核心事实" IN content)
  
  Level 3: 含"我"的自我指涉语句（>5字符）
    → ["我的", "我想", "我计划", "我喜欢", "我决定", ...]

未命中 → 直接返回空字符串（节省 ~200-500ms）
```

**写入预判 - `should_update_memory()`**（四层过滤）：

```
输入: (question, answer)
输出: (should_update: bool, reason: str)

Level 1: 问题长度 < 4 字 → 跳过（太短无信息量）
Level 2: 回答长度 < 30 字 → 跳过（大概率是闲聊）
Level 3: 不含个人信息关键词 → 跳过
    → 检测范围：身份类、偏好类、决策/计划类、家庭/背景类
Level 4: 全部通过 → 执行 LLM 记忆提取
```

这种设计将无效的记忆操作减少了约 **70-85%**（取决于对话内容类型）。

### X.5 知识图谱模块

#### X.5.1 金融本体论设计

知识图谱采用**芯片供应链领域本体论**，定义了 9 种实体类型和 18 种关系类型：

**实体类型（EntityType 枚举）**：

| 实体类型 | 说明 | 示例属性 |
|----------|------|----------|
| Company | 公司（设计公司、整机厂、设备商） | name, code, sector, market_cap |
| Product | 产品/部件（GPU/CPU/存储芯片） | name, type, process_node |
| Foundry | 代工厂/产线 | name, location, capacity |
| Material | 原材料/设备（硅片/光刻胶） | name, type, supplier |
| Location | 地点（国家/城市/园区） | name, type, country |
| Sector | 行业/概念板块 | name, code, description |
| Event | 事件（停产/事故/政策） | name, type, date, impact |
| Person | 人物 | name, title, company |
| Asset | 资产（股票） | code, price |

**关系类型（RelationType 枚举，重点抽取供应链关系）**：

```
DESIGNS:        (Company)-[设计]->(Product)          如: 英伟达-设计->A100 GPU
PURCHASES:      (Company)-[采购]->(Product)          如: 苹果-采购->A100芯片
OUTSOURCES_TO:  (Company)-[委托代工]->(Foundry)       如: 苹果-代工->台积电
MANUFACTURES_FOR:(Foundry)-[代工]->(Company)        如: 台积电-代工->苹果
DEPENDS_ON:     (Foundry)-[依赖]->(Material)         如: 台积电-依赖->ASML光刻机
COMPETES_WITH:  (Company)-[竞争]->(Company)          如: 英伟达-竞争->AMD
BELONGS_TO:     (Company)-[属于]->(Sector)           如: 英伟达-属于->AI芯片板块
AFFECTED_BY:    (Company)-[受影响]->(Event)          如: 台积电-受影响->地震
SUPPLIES:       (Company)-[供应]->(Company)          如: ASML-供应->台积电
LOCATED_IN:     (Entity)-[位于]->(Location)          如: 台积电-位于->台湾台南
```

#### X.5.2 LLM 驱动的实体/关系抽取

系统使用 `EntityExtractor` 类实现基于 LLM 的信息抽取。给定一段文本（如新闻稿），LLM 按照 predefined JSON Schema 输出实体和关系：

**抽取 Prompt 设计要点**：

1. 明确限定领域范围："只抽取与芯片/半导体/电子供应链相关的实体和关系"
2. 优先级指引："优先抽取：代工关系、供应链依赖、竞争关系"
3. 属性填充要求："关系属性尽量填充（工艺节点、占比、依赖程度）"
4. 结构化输出约束："只输出标准 JSON 格式"

```python
EXTRACTION_PROMPT = """你是一个专业的半导体/芯片供应链知识图谱抽取系统。
请从文本中抽取实体和关系，用于金融领域的上下游风险分析。

## 本体论
节点类型: Company, Product, Foundry, Material, Location, Sector, Event
关系类型（重点抽取）:
- DESIGNS, PURCHASES, MANUFACTURES_FOR, OUTSOURCES_TO,
  DEPENDS_ON, LOCATED_IN, COMPETES_WITH, SUPPLIES, BELONGS_TO, AFFECTED_BY

## 输入文本: {text}

## 输出格式 (JSON)
{"entities": [{"name":"实体名","type":"类型","properties":{}}],
 "relations": [{"source":"A","type":"关系","target":"B","properties":{}}]}
"""
```

抽取结果通过 Cypher 的 `MERGE` 语句写入 Neo4j（幂等写入，避免重复节点）：

```python
# 实体写入（MERGE 保证唯一性）
MERGE (e:{entity_type} {{name: $name}})
SET e += $properties

# 关系写入（模糊匹配节点名称）
MATCH (source:{source_type})
WHERE toLower(source.name) CONTAINS toLower($source_name)
MATCH (target:{target_type})
WHERE toLower(target.name) CONTAINS toLower($target_name)
MERGE (source)-[r:{relation_type}]->(target)
SET r += $properties
```

关系写入时采用 **模糊匹配**（`CONTAINS` 而非精确相等），容忍 LLM 抽取时实体名称的微小差异（如"台积电" vs "台湾积体电路"）。

#### X.5.3 Cypher 查询构建

`CypherQueryBuilder` 类封装了常用的图查询模板：

| 查询方法 | 功能 | 应用场景 |
|----------|------|----------|
| `find_company_relationships()` | 公司关系网络 | "XX公司的竞争对手有哪些？" |
| `find_supply_chain()` | 完整供应链 | "XX公司的上下游有哪些？" |
| `find_risk_impact()` | 代工厂风险传导 | "台积电停产会影响哪些公司？" |
| `find_competitors()` | 同行业竞争对手 | "AI芯片领域有哪些玩家？" |
| `find_sector_companies()` | 板块内公司列表 | "科技行业有哪些上市公司？" |
| `find_impact_chain()` | 影响链路推理 | "为什么XX会受影响？" |

查询均支持**模糊匹配**（`toLower(name) CONTAINS toLower($param)`），适应自然语言查询中的名称变体。

#### X.5.4 混合检索（KG + Vector）

`hybrid_retrieval()` 函数实现了知识图谱与向量检索的结果融合：

```python
def hybrid_retrieval(question, vector_results=None, top_k=5):
    # 1. KG 检索
    kg_result ← retriever.query(question)
    
    # 2. 结果融合
    combined_context = []
    IF kg_result.entities 非空 THEN
        kg_context ← 格式化实体列表 + 解释
        combined_context.append(kg_context)
    END IF
    
    IF vector_results 非空 THEN
        vector_context ← 格式化文档片段
        combined_context.append(vector_context)
    END IF
    
    RETURN {context: 合并结果, sources: 来源列表}
```

两种检索方式互补：KG 提供结构化的**实体-关系-实体**链路，Vector 提供非结构化的**文档片段**语义匹配。

### X.6 智能触发系统

#### X.6.1 设计目标

触发系统是整个系统的**调度中枢**，负责分析用户问题并决定激活哪些处理模块。其设计目标是：

1. **精准触发**：只在需要时激活对应模块，避免资源浪费
2. **可配置**：触发条件可通过环境变量调整
3. **可扩展**：新增模块只需注册新的 Trigger

#### X.6.2 双模触发机制

系统支持两种触发模式：

**模式一：关键词匹配（默认）**

每种模块类型维护一个关键词列表，当用户问题包含任一关键词时触发：

```python
class KeywordTrigger(BaseTrigger):
    DEFAULT_KEYWORDS = {
        "finance_akshare": ["股票", "股价", "基金", "指数", "大盘", "涨跌", ...],
        "quant": ["量化", "回测", "策略", "均线", "RSI", "MACD", 
                   "机器学习", "xgboost", "热启动", "快照恢复", ...],
        "resset": ["年报", "季报", "政府工作报告", "研究报告", "股吧", ...],
        "kg": ["关系", "影响", "关联", "竞争", "产业链", "上下游", ...],
        ...
    }
    
    def check(self, question):
        matched ← [kw for kw in keywords if kw.lower() in question.lower()]
        IF matched THEN
            confidence ← min(0.5 + 0.1 * len(matched), 1.0)
            RETURN TriggerResult(confidence=confidence, reason=f"匹配到: {matched}")
        END IF
        RETURN None
```

置信度计算公式：$confidence = \min(0.5 + 0.1 \times N_{matched}, 1.0)$，即每多匹配一个关键词置信度增加 0.1，上限 1.0。

**模式二：LLM 智能识别（可选）**

启用环境变量 `LLM_TRIGGER_ENABLED=true` 后，系统使用 LLM 自动分析问题应触发的模块组合：

```python
LLM_TRIGGER_PROMPT = """根据用户问题，判断需要触发哪些处理模块：
可用触发类型：hyde, reranker, finance_akshare, finance_tushare,
                memory, compliance, kg, general
输出JSON格式：{trigger_type, confidence, params, reason}
"""
```

LLM 模式的结果会与关键词模式**合并**：若两者都检测到同一类型触发，取更高置信度；LLM 还可补充关键词模式遗漏的触发。

### X.7 数据采集与定时调度

#### X.7.1 定时任务调度器（KGScheduler）

`KGScheduler` 基于 Python `schedule` 库实现，支持三种调度粒度：

| 调度类型 | 配置参数 | 适用场景 |
|----------|----------|----------|
| `daily` | hour, minute | 每日固定时间（如开盘前更新数据） |
| `hourly` | - | 每小时执行（如新闻爬取） |
| `interval_minutes` | minutes | 自定义间隔（如每30分钟） |

调度器运行在**后台守护线程**中，不阻塞主进程：

```python
class KGScheduler:
    def start(self, blocking=False):
        self.running = True
        IF blocking THEN
            self._run_loop()      # 阻塞式（用于测试）
        ELSE
            thread ← Thread(target=self._run_loop, daemon=True)
            thread.start()         # 后台运行
        END IF
    
    def _run_loop(self):
        WHILE self.running:
            schedule.run_pending()
            sleep(1)              # 每秒检查一次
```

每个任务执行时会记录**耗时、状态、错误信息**，历史记录持久化到 `logs/scheduler_logs/task_history.json`（最多保留100条）。

#### X.7.2 预定义任务

系统预设了三个定时任务（通过 `setup_default_tasks()` 注册）：

**任务1：`crawl_news` — 新闻爬取 + 实体抽取（每小时）**

```
流程：爬取财经新闻 → 过滤芯片/半导体相关 → LLM抽取实体/关系 → 写入Neo4j

详细步骤：
1. NewsCrawler.crawl_all(max_count_per_source=20, filter_relevant=True)
   → 从多个财经网站爬取新闻，本地缓存避免重复
2. EntityExtractor.extract(news_text)
   → 调用 DeepSeek LLM 抽取实体和关系
3. KGWriter.write_company/product/foundry/material/...
   → 按实体类型分别写入 Neo4j

   → 写入关系（含工艺节点、占比等属性）
```

**任务2：`update_stocks` — 股票基础信息更新（每日09:30）**

从 AKShare 获取A股上市公司列表，批量导入 Neo4j 的 Company 节点（限制100家/次）。

**任务3：`update_sectors` — 行业分类更新（每日09:00）**

获取申万行业分类数据，构建 Company-BELONGS_TO-Sector 关系。

#### X.7.3 知识图谱写入器（KGWriter）

`KGWriter` 封装了对 Neo4j 的所有写操作，采用 **MERGE 幂等写入**策略——同一实体/关系重复写入不会产生重复节点：

```python
# 公司节点写入示例
MERGE (c:Company {code: $code})   # 以股票代码为主键
SET c.name = $name,
    c.sector = $sector,
    c.market_cap = $market_cap,
    c.updated_at = datetime()

# 关系写入（模糊匹配 + 属性支持）
MATCH (source:Company) WHERE toLower(source.name) CONTAINS toLower($source_name)
MATCH (target:Sector) WHERE toLower(target.name) CONTAINS toLower($target_name)
MERGE (source)-[r:BELONGS_TO]->(target)
SET r += $properties, r.updated_at = datetime()
```

支持的节点写入方法：
- `write_company()` — 公司
- `write_sector()` — 行业板块
- `write_person()` — 人物（高管）
- `write_product()` — 产品/部件
- `write_foundry()` — 代工厂
- `write_material()` — 原材料/设备
- `write_location()` — 地点
- `write_relation()` — 关系（通用方法，支持属性）

批量操作方法：
- `import_companies_batch(companies)` — 批量导入公司
- `import_sectors_batch(sectors)` — 批量导入行业
- `build_company_sector_relations(map)` — 批量建立公司-行业关系

#### X.9.2 延迟加载与容错设计

系统大量采用**延迟加载（Lazy Initialization）**模式：

| 组件 | 加载时机 | 容错策略 |
|------|----------|----------|
| Ollama Embeddings | 首次向量检索时 | 未安装则提示 |
| Elasticsearch 连接 | 首次检索时 | 未启动则报错退出 |
| BGE-Reranker | 首次精排时 | sentence-transformers 未安装则跳过精排 |
| Neo4j 连接 | 首次 KG 查询时 | 未连接则 KG 功能禁用 |
| DeepSeek LLM | 首次生成时 | API Key 未配置则报错 |
| MemoryManager | 首次记忆操作时 | 目录自动创建 |
| ComplianceChecker | 首次合规检查时 | 未配置则跳过 |

每个模块都通过 `try-except` 包装导入和初始化过程，确保单一组件不可用时系统仍能降级运行。

#### X.9.3 关键性能优化措施

| 优化点 | 技术 | 效果 |
|--------|------|------|
| Embedding 计算 | LRU 缓存层（max_size=500） | 缓存命中率 60-80%，减少 Ollama 调用 |
| 记忆读取 | 双层触发判断 | 减少 70-85% 无效检索 |
| 记忆写入 | 四层预判过滤 | 减少 70-85% 无效 LLM 调用 |
| 向量检索 | 先粗排（k=10）后精排（top_k=3） | 平衡召回率和精度 |
| 组件加载 | 全局单例 + 延迟初始化 | 启动时间 < 2s |
| BM25 索引 | 按需重建（脏标记机制） | 仅在记忆变更后重建 |
| 新闻爬取 | 本地缓存（JSON 文件） | 避免重复爬取同一条新闻 |

### X.10 本章小结

本章详细阐述了 FinRAG-Advisor 系统的各核心模块的实现细节：

1. **Agent 智能体**：采用 Function Calling 两轮推理机制，将四类检索能力封装为工具，由 LLM 按需调用
2. **RAG 管道**：基于 LangGraph 编排，整合 HyDE 假设性文档增强、BGE-Reranker 精排、多源上下文融合和锐思 LLM 自主决策调用
3. **记忆系统**：OpenClaw 风格四层架构，BM25+向量混合检索（7:3加权）、MMR去重、时间衰减、LRU Embedding缓存、智能 Compaction 和双触发机制
4. **知识图谱**：芯片供应链领域本体论、LLM 驱动实体/关系抽取、模糊匹配 Cypher 查询、Pyvis 可视化
5. **触发系统**：双模触发（关键词+LLM），统一管理 9 种处理模块的激活条件
6. **数据采集**：定时调度器 + 新闻爬虫 + LLM 抽取 + Neo4j 写入的自动化 ETL 管道
